# 多智能体协作需求说明

- 文档状态：Draft v0.1
- 来源依据：`docs/drafts/drafts-001-agents-colabrate.md`
- 适用范围：nanobot 后续多智能体协作能力设计、评审与实现

## 1. 背景

当前 nanobot 已具备单智能体对话、工具调用、后台 `subagent` 执行、会话持久化和长期记忆等基础能力，但尚未形成一套正式的“主智能体规划 + 子智能体执行 + 结构化协作产物 + 结果回收”的协作协议。

草稿提出了“唯一真身 / 身外化身”两种工作模式。本说明将其整理为正式需求，并将“共享上下文”“共享记忆”等概念收敛为可实现、可测试、可追踪的系统要求。

## 2. 目标与非目标

### 2.1 目标

1. 支持单智能体模式与多智能体协作模式共存。
2. 在多智能体模式中，Main Agent 负责对人沟通、任务澄清、任务书维护、子任务编排与结果合并。
3. Sub Agent 负责在限定范围内独立执行任务，并以结构化回执单向 Main Agent 反馈结果。
4. 为协作过程建立统一产物：任务书、任务单、回执单、任务日志。
5. 为多智能体流程提供可审计、可恢复、可取消的执行闭环。
6. 在不破坏 nanobot 轻量化风格的前提下实现首版可用能力。

### 2.2 非目标

1. 首版不追求完整的通用工作流引擎或复杂 DAG 调度。
2. 首版不要求 Sub Agent 直接面对用户发言。
3. 首版不要求所有 Agent 对全局长期记忆拥有同等写权限。
4. 首版不要求引入新的重量级分布式组件。

## 3. 术语定义

| 术语 | 定义 |
| --- | --- |
| 唯一真身模式 | 仅使用一个 Agent 完成沟通、思考、工具调用和交付的模式。 |
| 身外化身模式 | 由 Main Agent 负责规划与协调，Sub Agent 负责执行具体子任务的模式。 |
| Main Agent | 面向用户的主智能体，负责澄清需求、生成任务书、分派任务单、接收回执单、汇总交付。 |
| Sub Agent | 由 Main Agent 派生的临时执行体，带着任务上下文和约束完成具体工作，完成后销毁。 |
| 任务书 | Main Agent 维护的主任务描述，是整个协作过程的事实源。 |
| 任务单 | Main Agent 发给某个 Sub Agent 的执行指令，描述边界、输入、输出和约束。 |
| 回执单 | Sub Agent 完成后提交给 Main Agent 的结构化执行结果。 |
| 任务日志 | Main Agent 与 Sub Agent 共同追加的协作事件日志，用于追踪过程、沉淀经验。 |
| 共享记忆 | 多智能体协作时可共同读取的任务相关上下文与长期记忆视图，不等价于“任何 Agent 都可随意改写全局记忆”。 |

## 4. 模式定义

### 4.1 唯一真身模式

唯一真身模式下，系统行为与当前 nanobot 单智能体工作流一致。系统直接由 Main Agent 完成理解、推理、工具调用和响应，不生成 Sub Agent，不创建任务单和回执单。

适用场景：

1. 单轮问答。
2. 简单工具调用。
3. 不需要并行执行或任务拆分的请求。

### 4.2 身外化身模式

身外化身模式下，Main Agent 不直接包办所有执行动作，而是将复杂任务拆为若干子任务，交由一个或多个 Sub Agent 处理。Main Agent 只保留用户沟通、规划、决策、检查和汇总职责。

适用场景：

1. 多步骤任务。
2. 可拆分并行的任务。
3. 需要隔离上下文、限制执行范围或分角色处理的任务。

## 5. 核心设计原则

1. Main Agent 是对外唯一接口，也是最终结果的唯一签发者。
2. 任务书是主事实源，任务单和回执单都必须能追溯到任务书。
3. Sub Agent 是临时执行体，默认不跨任务长期驻留。
4. 共享记忆采用“共享可见、主写从提议”的策略：Main Agent 负责全局记忆最终写入，Sub Agent 通过回执或任务日志提出增量信息。
5. “Sub Agent 具有与 Main 一样的上下文”在工程上定义为：获取同一任务书、必要历史摘要、相关长期记忆摘要、工具边界和任务约束；不要求无上限复制完整原始上下文。
6. 多智能体能力必须可关闭，不能破坏现有单智能体主路径。

## 6. 协作产物要求

### 6.1 任务书

任务书由 Main Agent 创建和维护，至少包含以下字段：

| 字段 | 要求 |
| --- | --- |
| task_id | 全局唯一 ID |
| title | 任务标题 |
| goal | 用户目标 |
| success_criteria | 成功判定标准 |
| constraints | 约束条件、禁止项、资源边界 |
| context_snapshot | 当前任务相关上下文摘要 |
| decomposition | 子任务拆分结果，需包含每个子任务的执行边界与资源控制信息 |
| resource_policy | 任务级资源控制策略，包括默认模型、并发上限、预算上限、超时和子任务覆写规则 |
| status | draft / running / blocked / completed / failed / cancelled |
| owner | Main Agent |
| created_at / updated_at | 时间戳 |

任务书中的子任务拆分不应只有“做什么”，还应包含“用什么资源做”。资源控制策略至少支持为子任务指定使用的模型；后续可扩展到 token 预算、执行超时、最大迭代次数、最大并发数和工具白名单。

### 6.2 任务单

任务单由 Main Agent 派发给 Sub Agent，至少包含以下字段：

| 字段 | 要求 |
| --- | --- |
| work_order_id | 全局唯一 ID |
| parent_task_id | 对应任务书 ID |
| assignee | Sub Agent 标识 |
| objective | 本次子任务目标 |
| inputs | 输入材料、文件、上下文片段 |
| allowed_tools | 允许使用的工具集合 |
| resource_profile | 本子任务的资源配置，如指定模型、预算、超时、最大迭代次数 |
| expected_output | 交付格式和验收口径 |
| dependencies | 前置依赖 |
| priority | 优先级 |
| status | pending / running / completed / failed / cancelled |

### 6.3 回执单

回执单由 Sub Agent 回传，至少包含以下字段：

| 字段 | 要求 |
| --- | --- |
| receipt_id | 全局唯一 ID |
| work_order_id | 对应任务单 ID |
| status | success / partial / failed |
| summary | 执行摘要 |
| outputs | 产出列表或结果引用 |
| evidence | 关键依据、命令结果、文件路径、链接等 |
| risks | 未解决风险和限制 |
| suggested_next_actions | 建议 Main Agent 采取的后续动作 |
| completed_at | 完成时间 |

### 6.4 任务日志

任务日志为追加写入的事件记录，至少记录以下信息：

1. 任务创建、派发、开始、完成、失败、取消。
2. Main Agent 的关键决策。
3. Sub Agent 的关键执行节点。
4. 对失败路径、已知坑点、规避策略的沉淀。

## 7. 功能需求

### FR-01 模式选择

系统必须支持单智能体模式和多智能体模式。Main Agent 应能够根据任务复杂度、并行性和上下文隔离需求判断是否进入多智能体模式。系统也应支持显式配置关闭多智能体能力。

### FR-02 用户沟通职责

Main Agent 必须是用户唯一可见的主沟通体，负责：

1. 理解并澄清用户需求。
2. 维护任务书。
3. 将系统中间过程转换成面向用户的自然表达。
4. 汇总多个 Sub Agent 的结果并给出最终交付。

### FR-03 任务拆解与派发

当任务进入多智能体模式时，Main Agent 必须能够：

1. 将任务拆分为可执行子任务。
2. 为每个子任务生成任务单。
3. 识别前置依赖和可并行关系。
4. 控制同时运行的 Sub Agent 数量。
5. 为每个子任务指定资源控制参数，如使用模型、预算和超时策略。

### FR-04 Sub Agent 生命周期

系统必须支持 Sub Agent 的完整生命周期：

1. 创建。
2. 上下文注入。
3. 执行。
4. 回传回执单。
5. 销毁。

Sub Agent 默认不保留跨任务身份，除非未来版本明确支持“常驻执行体”。

### FR-05 上下文继承

Sub Agent 必须获得完成任务所需的最小充分上下文，包括：

1. 对应任务单。
2. 对应任务书摘要。
3. 必要的用户约束和成功标准。
4. 相关长期记忆摘要。
5. 与任务直接相关的历史片段或文件引用。

系统不要求无差别复制 Main Agent 的完整消息历史，但必须确保 Sub Agent 在任务边界内具备独立完成工作的上下文。

### FR-06 共享记忆与写入策略

系统必须支持多智能体共享读取任务相关记忆，但写入策略必须分层：

1. Main Agent 可写任务书、任务日志和最终全局记忆。
2. Sub Agent 可写任务日志、回执单和任务级中间产物。
3. Sub Agent 对全局长期记忆的修改必须通过 Main Agent 合并，不得直接写入。

### FR-07 结果回收与合并

Main Agent 必须能够接收、解析并合并一个或多个 Sub Agent 的回执单，并基于回执状态决定：

1. 直接汇总交付。
2. 二次派发补充任务。
3. 中止任务并向用户说明原因。

### FR-08 可取消与可恢复

多智能体任务必须支持取消。取消主任务时，系统应同步取消该任务下未完成的 Sub Agent。系统应保留最小可恢复状态，至少能恢复任务书、任务单、回执单和任务日志。

### FR-09 错误处理

系统必须处理以下异常情况：

1. Sub Agent 执行失败。
2. 回执格式不完整。
3. 子任务超时。
4. 同一子任务重复执行。
5. Main Agent 在结果合并阶段失败。

系统应能记录错误，并给出重试、忽略、补派或终止的明确决策。

### FR-10 可观测性

系统必须提供任务级可观测能力，至少可以追踪：

1. 当前任务状态。
2. 每个任务单状态。
3. 每个 Sub Agent 的开始与结束事件。
4. 关键产物所在位置。

### FR-11 安全与权限边界

Main Agent 派发给 Sub Agent 的工具权限和算力资源都必须可控。系统应支持对 Sub Agent 使用的工具集合、模型选择、预算上限和执行时长进行限制，避免其默认继承全部高风险能力或无约束消耗资源。

## 8. 非功能需求

### NFR-01 轻量化

实现方案应优先复用 nanobot 已有的消息总线、会话、记忆和子智能体执行机制，避免引入与项目风格不匹配的重型基础设施。

### NFR-02 可审计

任务书、任务单、回执单和任务日志必须可落盘，并可供人工查看、回放和排查。

### NFR-03 可扩展

方案应允许后续演进到更复杂的任务图、审批流、模式路由和任务恢复机制。

### NFR-04 成本可控

Sub Agent 的上下文注入应遵循最小充分原则，避免无上限复制 Main Agent 的完整上下文导致成本失控。

### NFR-05 兼容现有能力

现有单智能体对话、工具调用、`spawn`、`/stop`、会话持久化与记忆固化能力不能因多智能体引入而退化。

### NFR-06 一致性

任务状态、回执状态、日志事件和最终用户响应之间必须可追溯，不能出现“用户看到完成，但系统内部仍为失败或未知”的状态分裂。

## 9. 验收场景

### AC-01 简单任务不启用多智能体

用户提出简单问答或单步操作时，系统保持唯一真身模式，不生成任务书和任务单。

### AC-02 复杂任务进入多智能体

用户提出可拆分的复杂任务时，Main Agent 创建任务书，拆解子任务，派发至少一个任务单，并在所有回执返回后给出统一结果。

### AC-03 子任务失败可被感知

当某个 Sub Agent 失败时，Main Agent 能识别失败回执，并决定重试、降级或终止，而不是静默忽略。

### AC-04 协作产物完整可查

任务结束后，任务书、任务单、回执单和任务日志均可在本地查看，且字段完整、相互可追溯。

### AC-05 取消行为一致

用户取消主任务后，关联 Sub Agent 被停止，任务状态更新为 cancelled，日志中有明确记录。

## 10. 后续演进方向

1. 引入显式模式路由，让系统自动在单智能体和多智能体之间切换。
2. 引入任务依赖图和检查点恢复。
3. 支持更细粒度的共享记忆检索。
4. 支持更丰富的 Sub Agent 角色模板与工具白名单策略。
